Cloudflare Workers Adapter Status

Understanding the Current State

Care Commons was originally built for Vercel's serverless functions, which run in a Node.js environment with full access to standard Node APIs. Cloudflare Workers, while conceptually similar, run in a different execution environment‚ÄîV8 isolates rather than containers‚Äîwhich means they don't have access to all Node.js built-ins by default.

The Workers adapter (api/worker.ts) bridges this gap, translating between Care Commons's Express-based API and the Workers runtime. It's functional and deployed in production, but we're continuously improving type safety and error handling. The core functionality works; we're refining the edges.

What Works Right Now

The Cloudflare deployment is production-ready and running for several agencies. All core features work:

  • Authentication and sessions ‚Äî Login, logout, password reset, OAuth flows
  • Scheduling and visits ‚Äî Creating schedules, checking in for visits, documenting care
  • EVV compliance ‚Äî GPS verification, state-specific validation
  • Client and caregiver management ‚Äî Full CRUD operations on all entities
  • Real-time updates ‚Äî Through Supabase's built-in real-time subscriptions
  • File uploads ‚Äî Clinical photos and documents via R2 storage

The infrastructure components are also solid:

  • wrangler.toml is configured with appropriate bindings for Hyperdrive, KV, and R2
  • GitHub Actions workflows handle automated deployments from main branch
  • The Supabase adapter (packages/core/src/db/supabase-adapter.ts) handles database operations cleanly

What We're Still Refining

Type safety could be tighter in a few areas. TypeScript catches most issues, but the adapter layer between Express and Workers sometimes requires type assertions that we'd prefer to eliminate. This doesn't affect functionality‚Äîthe code works correctly‚Äîbut better types would catch potential issues earlier during development.

Error handling is comprehensive but could be more granular. Right now, certain Worker-specific errors (like exceeding CPU time limits or KV quota) get caught and logged, but they could provide more helpful error messages to administrators. This is polish, not a functional gap.

We're also testing more edge cases around connection pooling with Hyperdrive. The vast majority of database operations work flawlessly, but under very high load (much higher than typical small-agency usage), some edge cases around connection recovery could be more robust.

How This Relates to the Express App

The original API (api/index.ts) is an Express application designed for Vercel. Express is a traditional Node.js web framework that expects a persistent server process. Workers don't work that way‚Äîeach request is handled in an isolated execution environment that starts and stops.

The Workers adapter wraps Express using Cloudflare's compatibility layer, which emulates enough of Node.js to make Express work. This is officially supported and works well, but it's not the theoretically optimal way to write Workers code. The "proper" approach would be to use Workers' native HTTP handling directly, without Express.

We use the Express compatibility layer deliberately. It means we maintain one codebase that deploys to both Vercel and Cloudflare, rather than maintaining two separate implementations. The performance difference is negligible‚ÄîWorkers with Express are still dramatically faster than traditional serverless functions‚Äîand the maintenance benefit is substantial.

Should You Deploy on Cloudflare?

Yes, absolutely. The Cloudflare deployment is production-ready and powers real agencies serving real clients. The "TODO" status is about continuous improvement, not fundamental incompleteness.

If you're starting fresh, we generally recommend Vercel for your first deployment because it's simpler‚Äîfewer services to configure, more integrated developer experience. But if you're expanding to multiple states or regions, or if you're cost-sensitive and will exceed Vercel's free tier, Cloudflare is an excellent choice that's fully supported.

What's on the Roadmap

Our improvement priorities for the Cloudflare adapter:

Near term (next few months):

  • Eliminate remaining type assertions in the adapter layer through improved TypeScript definitions
  • Enhanced error messages that surface Worker-specific issues more clearly
  • Comprehensive end-to-end testing specifically for Workers runtime edge cases
  • Documentation improvements based on real-world deployment experiences

Medium term (6-12 months):

  • Optional direct Workers HTTP handling for agencies that want maximum performance (while keeping Express as default for simplicity)
  • Improved observability with better integration into Cloudflare's analytics and logging
  • More sophisticated caching strategies using KV for frequently-accessed data

Long term (ongoing):

  • Continued refinement based on production feedback and edge cases discovered by agencies at scale
  • Performance optimizations as Cloudflare adds new Workers features and capabilities

Getting Involved

If you're deploying on Cloudflare and encounter issues, or if you have ideas for improvements, we'd love to hear from you. File issues on GitHub or reach out at support@neighborhoodlab.org.

If you're a developer comfortable with Workers and interested in contributing improvements to the adapter, check out api/worker.ts and the contributing guide. The codebase is well-documented and we're happy to mentor contributors.

See the full Cloudflare Deployment Guide for step-by-step instructions on deploying Care Commons to Cloudflare's edge network.
